# TaskCacheProcessor 派发逻辑说明

## 文档目的

本文件用于说明当前缓存任务派发逻辑，以及建议采用的优化方案。  
本次方案只聚焦 4 个点：

1. 缓存任务扫描频率
2. 普通缓存任务严格按 0/1/2/3 分规则排序
3. 每轮普通缓存任务的处理方式
4. 同巷道判断方式

另补充 1 条已确认的任务规则：

- `TaskIdentification = 1` 的任务，终点不受空满状态限制

## 一、当前逻辑概况

当前 `TaskCacheProcessor` 的核心流程可以概括为：

1. 每隔 1 分钟扫描一次缓存任务。
2. 如果有优先缓存任务（`Priority = 10`），先处理优先任务。
3. 普通缓存任务参考最近一条已完成正式任务的：
   - `TaskIdentification`
   - `ShelvesIdentification`
   进行排序。
4. 当前普通缓存任务的大方向排序接近下面这套规律：
   - 相反类型 + 同货架
   - 相同类型 + 同货架
   - 相反类型 + 不同货架
   - 相同类型 + 不同货架
5. 排完序后，会一次性选一批候选缓存任务进入本轮处理。
6. 每条候选任务真正下发前，还要通过点位锁定、业务校验、拆分校验等检查。

## 二、当前逻辑存在的问题

### 1. 扫描频率太慢

当前固定 1 分钟扫描一次，缓存任务响应太慢。

会出现下面这种情况：

- 点位已经释放；
- AGV 已经空闲；
- 但系统还没到下一轮扫描时间；
- 任务要继续等几十秒甚至接近 1 分钟才再次尝试。

这会放大现场对“任务卡住”“任务不下发”的感受。

### 2. 0/1/2/3 规律不是“硬规则”，而更像排序倾向

当前普通缓存任务虽然接近按这套规律排：

- 0 分：相反类型 + 同货架
- 1 分：相同类型 + 同货架
- 2 分：相反类型 + 不同货架
- 3 分：相同类型 + 不同货架

但代码层面不是显式用“分数”表达，所以：

- 逻辑不够直观；
- 后续排查不方便；
- 现场不容易解释；
- 一旦后面叠加过滤、批量截取，用户会感觉规律没有被严格执行。

### 3. 每轮一次拿太多任务，容易让规律“看起来失效”

当前思路更接近：

- 先按当前上下文排序；
- 一次性取出一批候选任务；
- 然后按顺序逐个尝试。

问题在于：

- 这批任务是在“同一个最近完成任务上下文”下排出来的；
- 如果前面很多任务恰好都属于同一方向，例如都偏上架；
- 那这一轮里就可能连续尝试很多同类任务。

现场表现就会像：

- 同时有上下架任务；
- 系统先连续做一批上架；
- 上架做完后，下一轮才轮到下架。

这也是客户感觉“之前优化逻辑失效”的重要原因之一。

### 4. 同巷道判断目前不够收敛

当前逻辑会混合使用：

- `ShelvesIdentification`
- 起点/终点前两位数字兜底

但现场现在已经明确：

- 任务都会带 `ShelvesIdentification`

因此继续依赖点位前两位，容易扩大阻塞范围，增加误判概率。

## 三、建议优化方案

## 1. 缓存任务扫描频率优化

### 建议逻辑

改成动态扫描：

- 有待处理缓存任务：每 3 秒到 5 秒扫描一次
- 没有待处理缓存任务：每 15 秒到 20 秒扫描一次

### 改动目的

- 降低缓存任务等待下一轮扫描的时间；
- 提高 AGV 空闲时的响应速度；
- 降低“库位已经空了但任务还不动”的现象。

### 效率提升

当前固定 1 分钟扫描时：

- 平均等待约 30 秒
- 最差接近 60 秒

优化后如果按 3 秒扫描：

- 平均等待约 1 到 2 秒
- 最差约 3 秒到 5 秒

保守估计，缓存任务响应速度可以提升约 10 倍以上。

## 2. 普通缓存任务改成显式 0/1/2/3 分规则

### 规则来源

每条普通缓存任务，都要和“最近一条已完成正式任务”对比两个字段：

- `TaskIdentification`
- `ShelvesIdentification`

其中：

- `TaskIdentification` 用于判断是否相反类型
- `ShelvesIdentification` 用于判断是否同货架/同巷道

### 明确评分规则

- 0 分：相反类型 + 同货架
- 1 分：相同类型 + 同货架
- 2 分：相反类型 + 不同货架
- 3 分：相同类型 + 不同货架

### 排序方式

普通缓存任务统一按下面顺序排序：

1. `Score ASC`
2. `CreateTime ASC`

### 说明

这里不是取消日期，而是：

- 先按 0/1/2/3 分决定优先级；
- 同分的任务只按创建时间排序。

也就是说，日期仍然保留，而且是同分时唯一的排序依据。

### 改动目的

- 让减少 AGV 空跑的规则更明确；
- 让现场和客户容易理解系统为什么先做哪一条；
- 避免现在“逻辑接近但不够直观”的问题。

### 效果

如果字段完整且准确，这套规则能更稳定地实现：

- 刚做完某货架出库，优先补同货架入库；
- 尽量减少 AGV 空跑和跨货架来回切换；
- 提高同货架连续作业效率。

## 3. 普通缓存任务按巷道维度逐条派发

### 当前问题

当前如果一轮按批量思路处理普通缓存任务，会出现：

- 这一轮里很多任务都按同一个上下文排序；
- 前面一串分数最低的任务可能都属于同方向；
- 结果现场看起来就是一边倒处理。

### 建议逻辑

普通缓存任务不再理解成“一轮批量放出很多条”，而是按巷道维度逐条处理：

- 先判断某个 `ShelvesIdentification` 是否已有执行中任务
- 如果该巷道已有执行中任务，则本轮跳过这个巷道
- 如果该巷道没有执行中任务，则从该巷道的缓存任务中按 0/1/2/3 分规则排序
- 该巷道本轮只尝试找到 1 条最合适且能成功下发的任务
- 一旦成功下发 1 条，这个巷道本轮立即停止继续派发
- 待这条任务完成后，下一轮再回到该巷道继续按规则寻找下一条最合适任务

### 为什么这样更合适

现场实际约束是：

- 一个巷道因为中转位原因，同一时刻通常只允许 1 条执行中任务；
- 因此同一个巷道，不需要一轮里连续成功派发多条任务；
- 更合理的方式是：巷道空闲时选 1 条最优任务派发，任务完成后再继续选下一条。

这里的关键不是“每轮总共取几条”，而是：

- 每个空闲巷道，每次只成功派发 1 条任务；
- 如果当前排序最靠前的任务因为点位锁定、拆分条件或业务校验失败，则继续尝试该巷道排序中的下一条任务；
- 直到这个巷道成功派发出 1 条任务，或者该巷道没有更多合适候选。

### 改动目的

- 保证同巷道任务严格按现场节奏一条一条推进；
- 让“相反类型优先、同货架优先”在每次巷道空闲后重新生效；
- 避免同巷道一口气成功派发多条任务导致排序规律被削弱。

### 效果

这项改动不能简单量化成几倍提升，但能明显改善：

- 只做上架、不做下架；
- 只做下架、不做上架；
- 同巷道任务一口气连续下发多条的现象；
- 让巷道在每次任务完成后重新选择当前最优缓存任务。

## 4. 同巷道判断统一使用 ShelvesIdentification

### 建议逻辑

同巷道统一定义为：

```text
ShelvesIdentification 相同 = 同巷道
```

不再依赖起点/终点前两位做正常调度判断。

### 改动目的

- 简化调度规则；
- 降低误判；
- 保证优先任务只阻塞真正同巷道的普通任务。

### 效果

优化后：

- 优先任务条件不满足时，只阻塞相同 `ShelvesIdentification` 的普通任务；
- 不同 `ShelvesIdentification` 的普通任务仍可继续派发；
- 避免把本来不该被挡住的任务一起卡住。

## 5. TaskIdentification = 1 的任务终点不看空满

### 规则说明

已确认业务口径如下：

- `TaskIdentification = 1` 表示入库/上架任务
- `fgHandover` 和 `shipment` 不会出现 `TaskIdentification = 1`
- 因此只要 `TaskIdentification = 1`，终点就不受空满状态限制

### 当前任务处理含义

这里的“不看空满”指的是：

- 系统在选择终点或自动分配终点时
- 仍然需要终点未锁定
- 但不再要求终点 `Quanitity` 为空、空字符串或 `0`

也就是说：

```text
TaskIdentification = 1
= 终点只看是否可锁定
不看空满状态
```

### 改动目的

- 避免入库/上架任务因为终点空满字段被误拦截；
- 保持入库任务终点选择更符合现场业务；
- 降低“明明应该能进，但因为空满字段没有清掉而被卡缓存”的概率。

### 影响范围

该规则适用于：

- `AddTaskAsync` 主流程中的终点自动分配
- 拆分任务类型 2 中第二段任务终点自动分配

### 效果

优化后：

- 入库/上架任务不会再因为终点空满状态被自动分配逻辑过滤掉；
- 终点是否可用主要取决于是否未锁定、是否能成功占用；
- 更贴近你现场已经确认的业务规则。

## 四、优化前后对比

| 项目 | 当前逻辑 | 优化后逻辑 |
|---|---|---|
| 扫描频率 | 固定 1 分钟一轮 | 有任务 3-5 秒，无任务 15-20 秒 |
| 任务排序表达 | 通过 SQL 排序倾向实现 | 直接显式按 0/1/2/3 分排序 |
| 日期作用 | 排序条件之一 | 同分时只按日期排序 |
| 每轮处理方式 | 一轮可能取较多候选任务 | 每个空闲巷道每次只成功派发 1 条最优任务 |
| 同巷道判断 | `ShelvesIdentification` + 点位前两位兜底 | 只用 `ShelvesIdentification` |
| 优先任务阻塞范围 | 可能偏宽 | 只阻塞真正同巷道普通任务 |
| 入库任务终点限制 | 终点会受空满状态影响 | `TaskIdentification = 1` 时终点不看空满 |

## 五、建议最终落地顺序

建议按下面顺序实施：

1. 先改扫描频率
2. 再把普通缓存任务排序改成显式 0/1/2/3 分，同分只按 `CreateTime`
3. 再改成按空闲巷道逐条派发普通任务
4. 最后统一同巷道判断只用 `ShelvesIdentification`
5. 再补充 `TaskIdentification = 1` 终点不看空满的规则说明

## 六、最终结论

本次建议方案不追求复杂调度，只做 4 个核心优化：

1. 扫描更快
2. 规则更明确
3. 每轮处理更贴近现场巷道节奏
4. 同巷道判断更准确
5. 入库任务终点规则更符合现场

其中最确定的效率提升是：

- 缓存任务响应速度预计提升约 10 倍以上

而在业务效果上，最明显的改进会是：

- 优先任务更快响应
- 普通任务更严格按减少空跑规律处理
- 上下架任务不会那么容易出现“一边倒”
- 不同巷道任务不再被无关优先任务误挡住
